## Lo cerrado de #1 (visor web) en esta sesión

Se creó la **sección 16. Visor web de memorias** en el PRD, más ajustes en resolución de contexto:

- **VIS-1** — el visor lee **exclusivamente del local** (el local replica toda la org, SYNC-5; búsqueda siempre local). Cero consulta viva al cloud.
- **VIS-2** — entrada por **selector de org** (A, B… + pseudo-org **LOCAL**). Aislamiento CFG-4: una org por vez.
- **VIS-3** — drill-down **org → proyecto → repo**.
- **VIS-4** — vista de proyecto = **topología** (repos + roles + aristas, DM-4/DM-5), no lista plana. Decisión **visual, no de datos** (la topología ya está almacenada). MVP estructurado → grafo después, sin migración.
- **Principio de evolución del visor** — mientras el dato esté en el modelo, MVP simple y mejoras = puro front, sin cruces ni migración.
- **VIS-5** — detalle de memoria con relaciones **inline** (tipo+dirección+título, click para saltar; datos de SRCH-5) + **"ver historial"** (cadena `replaced`, DM-13). Grafo de memorias = mejora futura.
- **VIS-6** — búsqueda **auto-scope con escape** para ensanchar. Ya no "adivina" porque el contexto es **determinístico** (RES-7/RES-8/CFG-3).
- **VIS-7** — acciones de curación sobre memorias propias (ID-10/ID-12): editar contenido (CAP-12), tag, relaciones (CAP-14). Borrar NO (admin only, DM-16).
- **VIS-8** — **re-anclaje**: cruzar de org IMPOSIBLE (CFG-4); dentro de la org solo el autor (repo→repo en mismo proyecto, o proyecto A→B); **límite 30 días** desde `created_at`.

Resolución de contexto (base que destrabó el auto-scope):

- **RES-7** — identidad de repo = git **`origin`, obligatoria siempre** (org y personal/LOCAL por igual). Sin origin (git init sin remoto) = no trabajable.
- **RES-8** — asignación de proyecto. Fuentes: mapping org-level (admin lo declara en cloud, baja por sync, keyed por origin) y config local. **Precedencia: gana el mapping de la ORG** sobre la config local. Orden: mapping org > config local > nombre carpeta (nunca autoritativo).
- **CFG-3 reescrito** — existe **exactamente UNA** org sin URL = pseudo-org **LOCAL**. Toda otra org DEBE tener URL cloud + credencial. Org ⇔ existe cloud (no "tiene acceso o no"); org con URL pero sin acceso (key revocada) sigue siendo org (gap #4).

## ⚠️ DECISIÓN PENDIENTE — retomar por acá

Estábamos cerrando el **#3 del visor (acciones de curación)**, en el flujo de **resolución de conflictos**, y apareció una **inconsistencia real del PRD** que el usuario detectó. Está **PROPUESTA, esperando confirmación del usuario para reescribir.**

**La inconsistencia:** SYNC-9 (líneas ~569-572) define el conflicto como del **mismo `id`** (mismo autor, multi-device) y dice que crea una relación **`caused` automática** hacia la ganadora (DM-12). Pero **`caused` (DM-10) relaciona dos memorias DISTINTAS** — no podés tener un `caused` de una memoria hacia sí misma. Con `id` igual, ese `caused` no tiene sentido. El `caused`-on-conflict nació para el caso de **dos memorias distintas** que chocan (mismo anclaje+topic+tag); cuando en el gap #3 acotamos SYNC-9 al mismo `id`, quedó arrastrando un mecanismo que solo servía para el otro caso.

**La separación correcta (dos cosas que NO son la misma):**

1. **Conflicto real (mismo `id`)** — dos ediciones divergentes de la **misma** memoria (multi-device SYNC-9, o local-vs-pull SYNC-12). Resolución = **reconciliar contenido** (mergear/elegir versión). **Sin `caused`.** Esto es lo que el visor sí debe contemplar.
2. **Redundancia (ids distintos)** — dos memorias con mismo anclaje+topic+tag. **No es conflicto**: es **supersesión** vía `replaces` (lo intenta el agente en captura, gap #6, o lo crea el dev curando). Sin `lifecycle=conflict` ni `caused`.

**Cambios propuestos (esperando OK del usuario):**

1. **DM-12 / SYNC-9** — sacar el `caused` automático; `lifecycle=conflict` se reserva para el **mismo `id`**.
2. **CAP-13** (gap #6) — su *backstop* del `replaces` se apoyaba en la detección de "conflicto" del caso ids-distintos. **Re-apuntarlo**: el backstop pasa a ser una **vista de "posibles duplicados"** en el visor (agrupa por anclaje+topic+tag y ofrece crear el `replaces`), no un estado de conflicto.
3. **Flujo del visor** — una sola **resolución de conflicto** (mismo `id`, reconciliar contenido) + una **vista aparte de "posibles duplicados"** para la supersesión.

**Última pregunta hecha al usuario (sin responder aún):** "¿Reescribo DM-12, SYNC-9 y CAP-13 con esa separación, y después cierro el flujo del visor?"
