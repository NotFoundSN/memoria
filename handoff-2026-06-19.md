## Lo cerrado de #1 (visor web) en esta sesión

Se creó la **sección 16. Visor web de memorias** en el PRD, más ajustes en resolución de contexto:

- **VIS-1** — el visor lee **exclusivamente del local** (el local replica toda la org, SYNC-5; búsqueda siempre local). Cero consulta viva al cloud.
- **VIS-2** — entrada por **selector de org** (A, B… + pseudo-org **LOCAL**). Aislamiento CFG-4: una org por vez.
- **VIS-3** — drill-down **org → proyecto → repo**.
- **VIS-4** — vista de proyecto = **topología** (repos + roles + aristas, DM-4/DM-5), no lista plana. Decisión **visual, no de datos** (la topología ya está almacenada). MVP estructurado → grafo después, sin migración.
- **Principio de evolución del visor** — mientras el dato esté en el modelo, MVP simple y mejoras = puro front, sin cruces ni migración.
- **VIS-5** — detalle de memoria con relaciones **inline** (tipo+dirección+título, click para saltar; datos de SRCH-5) + **"ver historial"** (cadena `replaced`, DM-13). Grafo de memorias = mejora futura.
- **VIS-6** — búsqueda **auto-scope con escape** para ensanchar. Ya no "adivina" porque el contexto es **determinístico** (RES-7/RES-8/CFG-3).
- **VIS-7** — acciones de curación sobre memorias propias (ID-10/ID-12): editar contenido (CAP-12), tag, relaciones (CAP-14). Borrar NO (admin only, DM-16).
- **VIS-8** — **re-anclaje**: cruzar de org IMPOSIBLE (CFG-4); dentro de la org solo el autor (repo→repo en mismo proyecto, o proyecto A→B); **límite 30 días** desde `created_at`.

Resolución de contexto (base que destrabó el auto-scope):

- **RES-7** — identidad de repo = git **`origin`, obligatoria siempre** (org y personal/LOCAL por igual). Sin origin (git init sin remoto) = no trabajable.
- **RES-8** — asignación de proyecto. Fuentes: mapping org-level (admin lo declara en cloud, baja por sync, keyed por origin) y config local. **Precedencia: gana el mapping de la ORG** sobre la config local. Orden: mapping org > config local > nombre carpeta (nunca autoritativo).
- **CFG-3 reescrito** — existe **exactamente UNA** org sin URL = pseudo-org **LOCAL**. Toda otra org DEBE tener URL cloud + credencial. Org ⇔ existe cloud (no "tiene acceso o no"); org con URL pero sin acceso (key revocada) sigue siendo org (gap #4).

## ✅ CERRADO 2026-06-19 — modelo de conflictos vs. identidad

La inconsistencia del `caused`-on-conflict se resolvió, y de paso apareció (la trajo el usuario) un **tercer caso** que faltaba separar. Quedaron **tres casos distintos**, antes mezclados:

1. **Conflicto de edición (mismo `id`, misma memoria)** — multi-device (SYNC-9) o local-vs-pull (SYNC-12). Resolución = **reconciliar contenido** (mergear/elegir versión). **Sin `caused`** (relacionaría una memoria consigo misma, violando DM-10). → DM-12, SYNC-9, **VIS-9** (nuevo).
2. **Redundancia (ids distintos, mismo anclaje+topic+tag)** — **no es conflicto**: posible supersesión vía `replaces`. El visor la **sugiere como posibles duplicados** (**VIS-10** nuevo), candidatas de baja confianza. **El código NUNCA infiere ni crea el `replaces`:** mismo anclaje+topic+tag es coincidencia estructural, no semántica (podrían ser `extends` o no tener relación). El juicio semántico vive en el **agente en captura** (CAP-14) o el **dev mirando el contenido** en el visor, que puede crear `replaces`, crear `extends` o descartar. CAP-13 re-apuntado a VIS-10 (antes apuntaba a DM-12).
3. **Colisión accidental de `id` (mismo `id`, memorias DISTINTAS)** — problema de **identidad**, no de contenido. El cloud re-asigna el UUID, discriminando por **`CREATE` vs `UPDATE`**: `CREATE` con id existente = colisión → re-asignar y avisar al local; `UPDATE` con id existente = conflicto de edición → caso 1. → **SYNC-14** (nuevo) + nota en DM-15.

**Decisión de fondo:** se mantuvo **single UUID v7** (no doble id local/cloud estilo Engram). La unicidad se garantiza con **validación server-side** al recibir; el doble id obligaría a reescribir aristas en cada subida. Con UUID v7 la identidad es estable desde el nacimiento y solo se reescriben aristas en la colisión accidental (≈ nunca).

**Ediciones aplicadas al PRD:** DM-15 (campo `id`), DM-12, SYNC-9, SYNC-14 (nuevo), CAP-13, VIS-9 + VIS-10 (nuevos), header (log sesión 2026-06-19 + correcciones de líneas que quedaban contradictorias en los logs del 17).

## ✅ CERRADO 2026-06-19 — estados pasivos del visor (VIS-11)

Tres indicadores que el visor muestra:
- **(a) Último sync (NFR-9):** fecha del último sync OK, pasivo sin alertas. Org LOCAL-only (CFG-3) → "local, sin cloud".
- **(b) Read-only vs editable (ID-10/ID-12):** badge visible. Editable = `author_id == -1` o `== mi id_dev`; el resto (otros devs por fan-out, memorias org cloud-side) = solo-lectura. La señal es visible ANTES de intentar editar.
- **(c) Draft (CAP-4):** **editable a mano** por el dev. Puede editar y mantener `draft`, o editar y **promover a `active`** a voluntad. Transición **unidireccional `draft → active`** (consolidada nunca vuelve a draft: sincroniza y puede estar linkeada → degradarla rompería el grafo). Promoción a mitad de sesión → el **agente abre un draft nuevo** y sigue.

**Principio de uso del visor (decisión de producto):** el visor es primariamente para **visualizar**; el canal principal de trabajo es el **agente**. El visor permite **curación mínima sobre memorias propias** (VIS-7), no es la herramienta de trabajo principal. Escrito en el encabezado de la sección 16.
